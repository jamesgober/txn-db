# txn-db v0.9.0 — Beta: the honest comparison

**Final benchmarks, gaps and all.** v0.9.0 is the beta in the run to 1.0. No
behavior or API change — it captures the final benchmark: a head-to-head against
the alternative a developer reaches for without MVCC, a single `RwLock<HashMap>`,
recorded honestly. From here to 1.0 is bug-fix and doc polish only.

## What is txn-db?

A multi-version concurrency control (MVCC) transaction engine: the layer that
turns a key-value store into a transactional database. Every write produces a new
version tagged with a commit timestamp, so readers get a stable snapshot without
ever blocking writers, and writers detect conflicts at commit time. The version
store is a trait, so the engine composes on top of any backend that can keep
timestamped versions.

## What's new in 0.9.0

### Measured against a single `RwLock<HashMap>`

txn-db is a transaction *layer*, not a full storage engine, so the fair
comparison is not against `sled` or `redb` but against the obvious hand-rolled
alternative: one reader-writer lock over a map. The new `benches/comparison.rs`
runs both over the same 10,000-key set.

| Workload | `RwLock<HashMap>` | `txn-db` |
|----------|------------------:|---------:|
| Point read, single thread | ~33 ns | **~26 ns** |
| Write, single thread | **~65 ns** | ~370 ns |
| Reads under a continuous writer, 2 readers | 4.7 M/s | **17 M/s** |
| … 8 readers | 15 M/s | **41 M/s** |
| … 32 readers | 19 M/s | **65 M/s** |

The story, told straight:

- **Reads are cheaper** in txn-db — a snapshot read returns a reference-counted
  `Arc<[u8]>`, while the lock-over-map clones the value out from under the lock.
- **Writes cost several times more** — the honest price of MVCC. A
  `RwLock<HashMap>` write is one lock and one insert; a txn-db commit takes a
  snapshot, allocates a timestamp, checks for conflicts, installs a new version,
  and advances the read watermark. That machinery is what buys snapshot
  isolation, conflict detection, and durability — none of which the bare map has.
- **Concurrent reads scale where the single lock cannot** — with one writer
  committing continuously, txn-db sustains roughly **3× the read throughput**,
  because a snapshot read never waits for a writer while every reader on the
  `RwLock` baseline blocks whenever the writer holds the exclusive lock.

If a workload is single-threaded and write-heavy with no need for transactions,
the bare `RwLock<HashMap>` is the right tool. The moment you need snapshot
isolation, conflict detection, or concurrent readers that do not stall behind
writers, the trade inverts. The full table, methodology, and the rest of the
hot-path and contention numbers are in
[`docs/PERFORMANCE.md`](https://github.com/jamesgober/txn-db/blob/main/docs/PERFORMANCE.md).

## Verification

Run on Windows x86_64 (Rust stable 1.95.x and MSRV 1.85) and on Linux (WSL2
Ubuntu); the configured CI matrix runs the same on Linux, macOS, and Windows:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
cargo test                       # default features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo +1.85 clippy --all-targets --all-features -- -D warnings
cargo +1.85 test --all-features
RUSTFLAGS="--cfg loom" cargo test --test loom_txn --release
cargo bench --bench comparison
cargo deny check
cargo audit
```

All green. The test suite is unchanged from 0.8 — 47 unit + 28 integration + 33
doctests + 3 `loom` model checks.
