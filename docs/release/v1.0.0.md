# txn-db v1.0.0 — Stable

**Stable.** The public API is frozen until 2.0, the durable commit-log format is
frozen for the 1.x series, and the Definition of Done is satisfied. txn-db is a
multi-version concurrency control transaction engine that has been through the
full arc — foundation, serializable isolation, durability, garbage collection,
optimization, hardening, and an honest benchmark — and is now ready to depend on.

## What is txn-db?

The layer that turns a key-value store into a transactional database. Every write
produces a new version tagged with a commit timestamp, so readers get a stable
snapshot without ever blocking writers, and writers detect conflicts at commit
time instead of holding locks for the lifetime of a transaction. It is a *layer*,
not a store: the version store is the `VersionStore` trait, so the engine composes
on top of any backend that keeps timestamped versions. An in-memory store ships.

## The shape of it

```rust
use txn_db::Db;

let db = Db::new();

// Lazy path — a single read or write needs no ceremony.
db.put(b"k".to_vec(), b"v".to_vec())?;
assert_eq!(db.get(b"k")?.as_deref(), Some(&b"v"[..]));

// Transactions when several operations must be atomic.
let mut tx = db.begin();
tx.put(b"a".to_vec(), b"1".to_vec());
tx.put(b"b".to_vec(), b"2".to_vec());
tx.commit()?;
# Ok::<(), txn_db::TxnError>(())
```

- **Snapshot isolation** by default; **serializable** (read-set validation that
  rejects write skew) under the `serializable` feature.
- **Sharded, lock-free commit path** — commits to unrelated keys do not contend;
  `loom`-checked.
- **Durable commit log** under the `durability` feature: `Db::open` logs each
  commit to a `wal-db` write-ahead log and syncs before acknowledging, then
  replays on restart.
- **Garbage collection** — `Db::collect_garbage` reclaims versions no live reader
  can observe, pinned by an oldest-reader watermark.
- **Pluggable backing store** — the `VersionStore` trait is a real seam, proven
  by an independent in-test implementation.

## What changed for 1.0

This is a stabilization release. The one substantive change is a **1-byte format
version** at the head of each durable commit record: the decoder accepts only
version 1 and rejects any other, so a future format cannot be misread as this
one. No published version carried persisted data, so there is nothing to migrate.
The record format is now specified normatively in
[`docs/COMMIT_LOG_FORMAT.md`](https://github.com/jamesgober/txn-db/blob/main/docs/COMMIT_LOG_FORMAT.md)
and frozen for 1.x.

Everything else is the frozen public surface: `Db` (`new` / `open` /
`with_store` / `begin` / `begin_serializable` / `snapshot` / `get` / `put` /
`delete` / `last_committed` / `collect_garbage` / `Clone` / `Default`),
`Transaction`, `Snapshot`, `Timestamp`, `TxnError`, `Result`, `VersionStore`,
`MemoryStore`, `WriteEntry`, and `prelude`.

## Definition of Done

Audited against the project standard:

- Compiles clean on Windows and Linux (WSL2), stable and MSRV 1.85; macOS via the
  CI matrix.
- `cargo fmt`, `cargo clippy --all-targets --all-features -D warnings`,
  `cargo test --all-features`, `cargo doc -D warnings` (default **and**
  all-features) all clean.
- `cargo audit` and `cargo deny check` pass.
- No `unwrap` / `expect` / `todo!` / `dbg!` in library code (clippy-enforced);
  **zero `unsafe`** (`#![forbid(unsafe_code)]`).
- The Tier-1 API headlines the README and `docs/API.md`; every public item has a
  runnable documentation example.
- Property tests cover the isolation and garbage-collection invariants; `loom`
  model checks cover the concurrent commit path and collection against a reader.
- Benchmarks track the hot paths with no regression; the contention curve and an
  honest comparison against `RwLock<HashMap>` are documented.

## Stability guarantees

- **Public API frozen until 2.0.** No existing item changes signature or
  behavior within 1.x; only additive, backward-compatible items may appear.
- **Durable commit-log format frozen for 1.x.** A 1.x reader reads any 1.x
  writer's log. A format change would ship under a new version byte and a major
  release.

```toml
[dependencies]
txn-db = "1.0"

# Opt into serializable isolation and/or a durable commit log:
txn-db = { version = "1.0", features = ["serializable", "durability"] }
```
